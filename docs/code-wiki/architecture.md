# 架构设计

## 系统架构

LT 导航站采用典型的 **Serverless 架构**，完全基于 Cloudflare 生态构建，无需传统服务器。

## 前端架构

### SvelteKit 路由结构

```
src/routes/
├── +page.svelte              # 首页（推荐列表）
├── +page.server.ts           # 首页服务端逻辑
├── +layout.svelte            # 全局布局
├── admin/
│   ├── +page.svelte          # 管理页面
│   └── +page.server.ts       # 管理页面服务端逻辑
└── api/
    └── metadata/
        └── +server.ts        # 元数据获取 API
```

### 组件与状态管理

项目使用 Svelte 原生状态管理，主要状态包括：

1. **数据状态**
   - `sites` - 所有推荐链接
   - `categories` - 所有分类
   - `settings` - 站点设置
   - `loggedIn` - 管理员登录状态
   - `user` - 当前登录用户（普通用户）

2. **UI 状态**
   - `activeCategory` - 当前选中分类
   - `query` - 搜索关键词
   - `darkMode` - 深色模式开关
   - `editMode` - 编辑模式开关
   - `showSiteEditor` - 链接编辑器显示状态
   - `showUserPanel` - 用户面板显示状态

### 响应式设计

项目采用 **移动优先** 的响应式设计，主要断点：
- `max-width: 1120px` - 平板设备
- `max-width: 680px` - 手机设备

## 后端架构

### 服务端模块

```
src/lib/server/
├── auth.ts           # 管理员认证
├── crm.ts            # CRM 同步
├── db.ts             # 数据库操作
└── user-auth.ts      # 用户认证
```

### 认证系统

系统包含两套认证体系：

#### 1. 管理员认证 (`auth.ts`)

```typescript
// 核心功能
- adminToken()  // 生成管理员 Token
- isAdmin()     // 验证管理员身份
- setAdminCookie()  // 设置管理员 Cookie
- clearAdminCookie() // 清除管理员 Cookie
- requireAdmin()    // 强制要求管理员权限
```

认证流程：
1. 用户输入密码
2. 服务端验证密码
3. 生成 SHA-256 哈希 Token
4. 设置 HttpOnly Cookie
5. 后续请求通过 Cookie 验证身份

#### 2. 用户认证 (`user-auth.ts`)

```typescript
// 核心功能
- hashPassword()       // 密码加密
- verifyPassword()     // 密码验证
- setUserCookie()      // 设置用户 Cookie
- currentRefUser()     // 获取当前用户
```

用户系统功能：
- 注册新用户
- 用户登录
- 更新联系方式
- VIP 开通
- CRM 同步

### 数据库操作层 (`db.ts`)

```typescript
// 核心功能
- dbFromPlatform()    // 从环境获取数据库实例
- listPublicSites()   // 获取公开推荐列表
- listCategories()    // 获取分类列表
- listAdminSites()    // 获取所有推荐（管理用）
```

降级策略：
1. 优先使用 D1 数据库
2. D1 不可用时回退到 `src/lib/sites.json` 静态数据
3. 提供 fallback 数据保证站点可用性

### CRM 集成 (`crm.ts`)

可选功能，用于与外部 CRM 系统同步用户数据：
- 注册时同步
- 登录时同步
- 更新联系方式时同步
- VIP 开通时同步

## 数据流

### 首页加载流程

```
用户访问 /
    ↓
+page.server.ts load()
    ↓
dbFromPlatform() 获取 D1 实例
    ↓
listPublicSites() 查询数据
    ↓
isAdmin() 检查登录状态
    ↓
返回数据给 +page.svelte
    ↓
渲染页面
```

### 编辑推荐流程

```
管理员点击「编辑」
    ↓
打开编辑器模态框，填写表单
    ↓
提交表单到 ?/updateSite action
    ↓
+page.server.ts update action
    ↓
验证管理员身份
    ↓
执行数据库 UPDATE
    ↓
返回成功响应
    ↓
刷新页面展示更新后内容
```

### 获取元数据流程

```
用户在编辑器中输入 URL
    ↓
点击「自动获取」按钮
    ↓
fetch('/api/metadata?url=...')
    ↓
+server.ts GET handler
    ↓
fetch 目标网站
    ↓
解析 HTML 提取 title/description/icon
    ↓
返回 JSON 数据
    ↓
自动填充编辑器表单
```

## 关键设计决策

### 1. 为什么选择 Cloudflare D1？

- **完全托管**：无需管理服务器、备份、扩容
- **低延迟**：数据存储在边缘网络
- **成本低**：免费额度充足
- **开发体验好**：SQLite 兼容，Wrangler 工具链完善
- **与 Pages 深度集成**

### 2. 为什么选择 SvelteKit？

- **编译时优化**：生成高效代码，运行时开销小
- **服务器端渲染 (SSR)**：SEO 友好，首屏加载快
- **API Routes**：无需额外后端框架
- **Edge 部署**：完美适配 Cloudflare Pages Functions
- **TypeScript 原生支持**

### 3. 降级设计

考虑到开发环境和可能的故障，系统设计了完善的降级方案：
- 静态 JSON 数据作为后备
- 本地开发不需要 D1
- CRM 功能可选配置

### 4. 安全性考虑

- 密码使用哈希存储（SHA-256 + salt）
- Cookie 设置为 HttpOnly、SameSite=Lax
- 生产环境强制 Secure Cookie
- 表单验证防止注入攻击
- 管理员权限验证在服务端执行
